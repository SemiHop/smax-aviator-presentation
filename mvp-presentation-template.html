Performance Optimization: OData API on Passive Database

The integration uses SMAX Out-of-Box (OOB) OData API which runs on the passive database. This architectural choice ensures zero impact on production SMAX performance - all read operations for data synchronization occur against the standby database, leaving the active database free to handle user transactions.

Key Architectural Principles

  • On-Premises Data Sovereignty: All SMAX data stays internal
  • Query-Time Security: RBAC enforced at vector search time
  • Session-Based Caching: User context cached for performance
  • Modular Design: Clean service separation for extensibility
  • Zero Production Impact: OData API on passive DB - minimal performance overhead

Four-Layer Security Model

Layer 1: Edge Security (DMZ)

  • OAuth 2.0 token validation from Azure AD
  • Rate limiting (100 req/min per user)
  • SSL/TLS termination
  • DDoS protection

Layer 2: Application Security

  • User context extraction from Azure AD token (email/UPN)
  • RBAC middleware enforces permissions on every request
  • Session-based access control caching (1-hour TTL)
  • API request validation and sanitization

Layer 3: Data Security

  • Data Domains: User must have access to at least one domain
  • Roles: Entity role is "Any" OR matches user's role
  • Locations: Entity location is "Any" OR matches user's location
  • Groups: Entity groups is "Any" OR user in allowed groups
  • Vector DB query-time metadata filtering
  • No data leakage through error messages

Layer 4: SMAX Security

  • Service account with read-only permissions
  • All data domain rules respected
  • All entitlement rules enforced
  • Comprehensive audit logging

User Context & Access Control Flow

How User Identity Flows Through the System

Step 1: User Authentication

User logs into Teams/Outlook via Azure AD. Copilot receives authenticated user token.

Step 2: Token Passed to On-Prem API

Azure AD token sent in Authorization header to FastAPI integration layer.

Step 3: Extract User Identity

FastAPI validates token with Azure AD and extracts user email/UPN.

Step 4: Check Cache

Look up user's access context in Redis cache (role, domains, location, groups).

Step 5: Fetch from SMAX (if cache miss)

Query SMAX APIs to get user's role, groups, location. Parse data domains from role permissions.

Step 6: Build Access Filter

Construct Qdrant metadata filter based on user's 4-dimensional access context.

Step 7: Execute Filtered Search

Vector search returns ONLY authorized results matching user's permissions.

Similar Ticket Identification & Tagging System

How Similar Tickets Are Found

The system uses a hybrid approach combining vector similarity search with intelligent tagging to find relevant tickets and provide accurate recommendations.

1. Vector Similarity (Primary)

  • Semantic Understanding: Embeddings capture meaning beyond keywords
  • Cosine Similarity: Measures how close ticket descriptions are
  • Example: "Email not syncing" matches "Outlook sync stopped"
  • Threshold: Similarity score > 0.75 (configurable)

2. Tag-Based Filtering (Secondary)

  • Category Tags: email, network, hardware, software
  • Component Tags: outlook, exchange, vpn, printer
  • Resolution Tags: cache-clear, reboot, config-change
  • Priority Tags: urgent, high, medium, low

Hybrid Search Algorithm

1 Vector Search: Find top 20 semantically similar tickets (cosine similarity)
2 Tag Boosting: Re-rank results by matching tags (boost score +0.1 per tag)
3 RBAC Filtering: Remove tickets from inaccessible data domains
4 Resolution Filtering: Prioritize resolved tickets with documented solutions
5 Return Top 5: Present most relevant tickets with resolution summaries

Backup & Disaster Recovery Plan

Comprehensive Backup Strategy

Qdrant Vector Database

  • Daily Snapshots: Full collection backup at 2 AM
  • Incremental Backups: Every 6 hours (changes only)
  • Retention: 30 days of daily backups
  • Storage: Separate NAS or S3-compatible storage
  • Compression: Gzip compression (70% size reduction)

Redis Cache

  • RDB Snapshots: Every 6 hours (save 900 1, save 300 10)
  • AOF (Append-Only File): Enabled for transaction replay
  • Retention: 7 days of snapshots
  • Note: Cache is reconstructable from SMAX

Application Configuration

  • Git Repository: All config files version controlled
  • Environment Variables: Backed up to secure vault
  • Docker Compose Files: Stored in Git with tags
  • NGINX Config: Version controlled + backup

Logs & Audit Trails

  • Application Logs: Shipped to ELK/Splunk daily
  • Retention: 90 days in log aggregator
  • Security Logs: 1 year retention for compliance
  • Archive: Compressed and moved to cold storage
